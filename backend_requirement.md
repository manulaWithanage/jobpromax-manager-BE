# Backend API Requirements

**Target Stack:** Python (FastAPI), MongoDB
**Frontend Codebase:** React (TypeScript)

## 1. Overview
The goal is to replace the current client-side mock implementation (`src/services/api.ts` and `src/lib/mockData.ts`) with a real RESTful API. The backend should expose endpoints to manage Tasks, Roadmap Phases, Feature Statuses, and Dashboard analytics.

## 3. API Endpoints

### 3.1. Tasks API
- **GET /tasks**
  - Returns a list of all tasks.
- **PATCH /tasks/{id}**
  - Updates a task (e.g., status change).
  - Body: `Partial<Task>`

### 3.2. Roadmap API
- **GET /roadmap**
  - Returns all roadmap phases.
- **POST /roadmap**
  - Creates a new roadmap phase.
- **PATCH /roadmap/{id}**
  - Updates a phase.
  - Body: `Partial<RoadmapPhase>`
  - *Note:* To implement `toggleDeliverable` from the frontend, the client can utilize this endpoint by sending the updated `deliverables` array.
- **DELETE /roadmap/{id}**
  - Deletes a phase.

### 3.3. Features API
- **GET /features**
  - Returns all features.
- **POST /features**
  - Creates a new feature.
- **PATCH /features/{id}**
  - Updates a feature status or details.
  - Body: `Partial<FeatureStatus>`
- **DELETE /features/{id}**
  - Deletes a feature.

## 4. Dashboard & Analytics API

### 4.1. KPI Stats API
**Collection Name:** `dashboard_stats` (Single document or calculated on fly)

- **GET /dashboard/kpi**
  - Returns the 3 key metrics: Overall Completion, Current Sprint info, and Velocity trends.
  - **Response Structure:** `KPI[]`

### 4.2. Pipeline Items API

## 5. Authentication & Authorization (via Clerk)

### 5.1. Overview
We will use **Clerk** (Next.js SDK) for authentication. The Python backend will **verify** the JWTs generated by Clerk but will not handle login logic or store password hashes.

### 5.2. Clerk Integration
- **Frontend**: Handles Login/Signup/Session via `@clerk/nextjs`.
- **Backend**:
  - Receives JWT in `Authorization: Bearer <token>` header.
  - Verifies token using Clerk's JWKS (JSON Web Key Set).
  - Extracts `sub` (User ID) and custom claims (Roles).

### 5.3. Role-Based Access Control (RBAC)
- **Roles:** Managed in Clerk Dashboard (Metadata) or a simple mapping in Backend.
- **Rules:**
  - **Manager:** Full access.
  - **Developer:** Can view all, update Tasks/Phases.
  - **Leadership:** Read-only access.

*(Note: The `users` collection is no longer required as Clerk manages user identities. We may optionally store a `users` collection just for application-specific profile data if needed, but not for auth credentials.)*


### 6.2. User Management API (New)
*Note: Since we are using Clerk, the backend must use the **Clerk Backend API** to perform these actions.*

- **GET /users**
  - **Access:** Manager Only.
  - **Behavior:** Fetches user list from Clerk API.
  - **Response:** List of users with their roles. Each user object should match:
    ```json
    {
      "id": "string",
      "name": "string",
      "email": "string",
      "role": "manager" | "developer" | "leadership",
      "isSuperAdmin": boolean
    }
    ```

- **POST /users**
  - **Access:** Manager Only.
  - **Body:** 
    ```json
    { 
      "name": "John Doe",
      "email": "john@example.com", 
      "role": "manager" | "developer", 
      "password": "..." (optional temporary) 
    }
    ```
  - **Behavior:** Creates a user in Clerk. Sets `public_metadata.role` to the specified role. Updates `first_name` and `last_name` based on input `name`.

- **DELETE /users/{id}**
  - **Access:** Manager Only.
  - **Behavior:** Deletes user in Clerk.
  - **Constraint (Super Admin):** 
    - The backend **MUST** check if the target user is a "Super Admin".
    - `Super Admin` status can be defined by a specific email in environment variables (e.g., `SUPER_ADMIN_EMAIL`) or a specific Clerk User ID.
    - If target is Super Admin, return `403 Forbidden`.

## 7. Implementation Guidelines (FastAPI)

1.  **Auth Library**: Use `fastapi.security.OAuth2PasswordBearer` and `python-jose`/`pyjwt` to verify Clerk tokens.
2.  **Clerk SDK**: Use the official `clerk-sdk-python` (if available) or raw HTTP requests to `https://api.clerk.com/v1` with `CLERK_SECRET_KEY`.
3.  **Pydantic Models**: Create Pydantic models that validate the enums strictly (e.g., `TaskStatus` enum).
4.  **ODM**: Use **Beanie** or **Motor** for async MongoDB interactions. 
5.  **CORS**: Enable CORS (`CORSMiddleware`) to allow requests from the React frontend (usually `http://localhost:3000`).
6.  **Response Format**: Ensure `_id` from MongoDB is converted to `id` (string) in the JSON response to match the frontend interfaces.
7.  **Calculated vs Stored**: For KPIs and Charts, decide if these should be stored as static documents (easier for MVP) or calculated dynamically from the `tasks` collection (better for real-world usage). For this migration, serving the static structures initially is acceptable.
